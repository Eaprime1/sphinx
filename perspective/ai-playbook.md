# AI Playbook

This page formalizes the conventions for AI-first authoring in this documentation set.
All AI partners and human authors follow these conventions so the collaboration is
transparent and the documentation is honest about how it was made.

## Front-matter on every page

Every `.md` page in this doc set (except stubs and indexes) carries a front-matter block:

```yaml
---
human_author: Perspective Project
ai_partner: Claude (Anthropic)
creation_mode: manual
---
```

`creation_mode` values:

| Value | Meaning |
|---|---|
| `manual` | Written by a human author. AI may have suggested edits but did not generate the text. |
| `generated` | Substantively generated by an AI partner from a prompt. Human reviewed and approved. |
| `sparklized` | The page has been compressed into a Sparkle. This is the final state. |

## Prompt blocks

When a section is substantially generated from a specific AI prompt, record the prompt
in a `{note}` admonition immediately after the section heading:

```markdown
## Gravity Well Cosmology

:::{note} Prompt used
"Explain the four types of gravity wells in the Perspective system —
Gravitar, Magnetar, Pulsar, and Void — focusing on their behavioral character
and lifecycle. Write for a reader who understands the architecture overview
but hasn't read the explanation pages yet."
:::
```

This turns the AI into a documented tool rather than a hidden influence.

## Perspective's notes

AI partners may leave commentary on structural choices, naming decisions, or areas of
uncertainty using `{admonition}` blocks labeled **Perspective's notes**:

```markdown
:::{admonition} Perspective's notes
I placed this section before the field list because readers arriving from the
explanation pages need orientation before precision. The field list alone would
be disorienting without the description paragraph.
:::
```

These blocks are part of the record. Do not remove them during editing — they can be
graduated to a `{note}` or removed when the page is sparklized.

## Term discipline

All {doc}`glossary` terms must use the `:term:` role when referenced:

```markdown
When content crosses the {term}`Aperture`, it becomes canonical.
```

This keeps the glossary wired to the text and ensures cross-references survive restructuring.

## Graduating content from play/

Content in `play/` is a Pool item. To graduate it to a stable quadrant:

- [ ] The page has a clear Diátaxis quadrant (tutorial / how-to / explanation / reference)
- [ ] Front-matter is complete (human_author, ai_partner, creation_mode)
- [ ] All glossary terms use `:term:` roles
- [ ] All cross-references use the correct Sphinx roles (`:doc:`, `:ref:`, `:term:`)
- [ ] The page builds without warnings (`make html`)
- [ ] A human author has reviewed the content
- [ ] The page is added to the appropriate `toctree`

This checklist is the Aperture for this documentation system.

## AI partner context packet

When starting a new authoring session with an AI partner, provide:

1. `intro/gravity-architecture.md` (system overview)
2. `glossary.md` (term vocabulary)
3. `ai-playbook.md` (this file — conventions)
4. The target page's location in the Diátaxis quadrant

This is the minimum context for an AI to contribute consistently.
